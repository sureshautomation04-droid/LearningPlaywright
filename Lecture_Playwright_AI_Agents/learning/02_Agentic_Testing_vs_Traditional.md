# Agentic Testing vs Traditional Testing

## Introduction

The testing landscape is shifting. For two decades, automated testing meant writing
deterministic scripts with fixed selectors, hardcoded data, and linear execution flows.
Agentic testing introduces a fundamentally different paradigm -- AI-driven agents that
explore, adapt, and self-heal.

This document breaks down the differences, trade-offs, and when to use each approach.

---

## Traditional Testing

Traditional test automation follows a **script-first** model:

1. A human writes test scripts manually
2. Tests use static selectors (CSS, XPath, data-testid)
3. Test data is hardcoded or loaded from fixtures
4. Tests break when the UI changes
5. Maintenance is a constant, manual effort

```typescript
// Traditional Playwright test
import { test, expect } from '@playwright/test';

test('login with valid credentials', async ({ page }) => {
  await page.goto('https://the-internet.herokuapp.com/login');
  await page.locator('#username').fill('tomsmith');
  await page.locator('#password').fill('SuperSecretPassword!');
  await page.locator('button[type="submit"]').click();
  await expect(page.locator('#flash')).toContainText('You logged into a secure area!');
});
```

**Problems with this approach:**
- If `#username` changes to `#user-email`, the test breaks
- If the success message text changes, the assertion fails
- No adaptability -- the test does exactly what was scripted, nothing more

---

## Agentic Testing

Agentic testing follows an **explore-first** model:

1. An AI agent receives a goal ("test the login page")
2. The agent explores the page by taking snapshots of the DOM
3. It identifies interactive elements and user flows
4. It generates test scenarios including edge cases
5. It writes test code based on real execution, not guessing
6. When tests break, a healer agent diagnoses and fixes them

The same login test, but generated by an agent, might look like:

```typescript
// Agent-generated Playwright test
import { test, expect } from '@playwright/test';

test.describe('Login Page Tests', () => {
  test('should login successfully with valid credentials', async ({ page }) => {
    // Navigate to the login page
    await page.goto('https://the-internet.herokuapp.com/login');

    // Fill in the username field
    await page.getByLabel('Username').fill('tomsmith');

    // Fill in the password field
    await page.getByLabel('Password').fill('SuperSecretPassword!');

    // Click the Login button
    await page.getByRole('button', { name: 'Login' }).click();

    // Verify successful login message is displayed
    await expect(page.getByText('You logged into a secure area!')).toBeVisible();
  });

  test('should show error for invalid credentials', async ({ page }) => {
    await page.goto('https://the-internet.herokuapp.com/login');
    await page.getByLabel('Username').fill('invaliduser');
    await page.getByLabel('Password').fill('wrongpassword');
    await page.getByRole('button', { name: 'Login' }).click();
    await expect(page.getByText('Your username is invalid!')).toBeVisible();
  });
});
```

**Key differences in the generated code:**
- Uses semantic locators (`getByLabel`, `getByRole`) instead of CSS selectors
- Includes both happy path and negative test cases
- Comments describe intent, making tests self-documenting
- Locators are more resilient to UI changes

## Comprehensive Comparison

| Dimension               | Traditional Testing          | Agentic Testing                  |
|--------------------------|------------------------------|----------------------------------|
| **Test Creation Speed**  | Hours to days per test suite | Minutes per test suite           |
| **Maintenance Effort**   | High -- manual updates       | Low -- self-healing agents       |
| **Selector Strategy**    | Static CSS/XPath/data-testid | Semantic (role, label, text)     |
| **Adaptability**         | None -- breaks on UI changes | Adapts via page snapshot         |
| **Coverage**             | Only what is scripted        | Discovers untested flows         |
| **Edge Case Discovery**  | Depends on human creativity  | AI generates edge cases          |
| **Initial Setup Cost**   | Low (just write tests)       | Medium (LLM API keys, config)   |
| **Running Cost**         | Minimal (CI compute only)    | Higher (LLM API calls)          |
| **Determinism**          | Fully deterministic          | Non-deterministic (LLM output)  |
| **Debugging**            | Clear stack traces           | Agent reasoning logs             |
| **CI/CD Integration**    | Native and mature            | Emerging support                 |
| **Expertise Required**   | Programming + testing skills | Prompt engineering + oversight   |
| **Trust Level**          | High -- predictable behavior | Requires validation              |
| **Best For**             | Critical path regression     | Exploratory and new feature tests|

---

## When to Use Traditional Testing

- **Critical business flows** that must be 100% deterministic
- **Regulatory compliance** where test evidence must be reproducible
- **Performance benchmarks** that require exact timing measurements
- **Simple CRUD operations** where writing a test is faster than prompting an agent
- **Existing mature test suites** that are well-maintained

## When to Use Agentic Testing

- **New feature exploration** where you need quick coverage
- **Large applications** where writing tests manually is impractical
- **Frequent UI changes** that break traditional selectors
- **Exploratory testing** to discover edge cases
- **Test suite bootstrapping** to generate an initial set of tests quickly

## The Hybrid Approach: AI-Assisted with Human Oversight

The most effective strategy combines both:

```
  ┌─────────────────────────────────────────────────┐
  │            HYBRID TESTING STRATEGY              │
  ├─────────────────────────────────────────────────┤
  │                                                 │
  │  Layer 1: CRITICAL TESTS (Traditional)          │
  │  - Login, checkout, payment                     │
  │  - Hand-written, fully deterministic            │
  │  - Human-maintained                             │
  │                                                 │
  │  Layer 2: FEATURE TESTS (Agent-Generated)       │
  │  - New features, UI flows                       │
  │  - Agent generates, human reviews               │
  │  - Agent heals when broken                      │
  │                                                 │
  │  Layer 3: EXPLORATORY (Fully Agentic)           │
  │  - Discovers new scenarios                      │
  │  - Runs periodically, reports findings          │
  │  - Human triages results                        │
  │                                                 │
  └─────────────────────────────────────────────────┘
```

**Workflow:**
1. Agent generates initial test suite from a URL
2. Human reviews and approves generated tests
3. Approved tests run in CI/CD like traditional tests
4. When tests fail, healer agent attempts fixes
5. Human reviews healer's fixes before merging
6. Agent periodically explores for new test opportunities

---

## Key Takeaway

Agentic testing does not replace traditional testing -- it augments it. The future of QA
is not "AI vs humans" but "AI with humans." Use agents to handle the tedious, repetitive
parts of testing while humans focus on strategy, edge cases that require domain knowledge,
and final validation.
